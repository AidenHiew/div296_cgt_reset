# Building the SPFx package (`div296-calculator.sppkg`)

This folder holds the **calculator-specific source** for a SharePoint Framework
(SPFx) web part. The surrounding SPFx project scaffolding (gulpfile, tsconfig,
`package.json`, `config/config.json`, etc.) is **generated by Microsoft's
official Yeoman generator** rather than hand-written here — that guarantees the
toolchain files match your installed SPFx/Node versions. You scaffold once, drop
these files in, and build.

> **Honesty note:** this package was **not compiled in the environment that
> authored it** (no SPFx toolchain there). The drop-in code is a faithful port
> of the verified `web/` site (`calcs.ts` is a copy of `web/calcs.js`, which
> `web/tests/parity.test.js` pins to the §12 numbers). `calcs.ts`/`ui.ts` carry
> `// @ts-nocheck` to avoid type friction on the JS port. Run `gulp build` after
> scaffolding to confirm a clean compile before packaging.

## What you provide (the drop-in files)

```
src/webparts/div296Calculator/
  Div296CalculatorWebPart.ts          web part wrapper (injects markup + boots UI)
  Div296CalculatorWebPart.manifest.json
  calcs.ts                            engine (copy of web/calcs.js)
  ui.ts                               scoped UI (port of web/app.js)
  markup.ts                           generated CSS + HTML (do not hand-edit)
config/package-solution.json          solution metadata (GUIDs pre-filled, no API perms)
build-spfx-markup.mjs                  regenerates markup.ts from web/
```

GUIDs in the manifest and `package-solution.json` are already unique — no need
to change them.

## Prerequisites

- **Node.js 18.x LTS** (match your SPFx version's supported Node — SPFx 1.18.x → Node 18).
- Global tools: `npm i -g gulp-cli yo @microsoft/generator-sharepoint`

## Steps

**1. Scaffold the project** (creates the toolchain files):

```bash
mkdir div296-spfx && cd div296-spfx
yo @microsoft/sharepoint
#  Solution name:        div296-calculator
#  Component type:       WebPart
#  Web part name:        Div296Calculator   (so the folder matches the drop-ins)
#  Framework:            No JavaScript framework
```

**2. Copy the drop-in files** from this folder over the generated stubs:

```bash
# from the repo root, with $SPFX pointing at the scaffolded project
cp docs/hosting/spfx/src/webparts/div296Calculator/{Div296CalculatorWebPart.ts,Div296CalculatorWebPart.manifest.json,calcs.ts,ui.ts} "$SPFX/src/webparts/div296Calculator/"
cp docs/hosting/spfx/config/package-solution.json "$SPFX/config/package-solution.json"
```

**3. Regenerate `markup.ts`** into the scaffolded project (keeps it in sync with
the live site):

```bash
# adjust the output path inside build-spfx-markup.mjs, or run then copy:
node docs/hosting/spfx/build-spfx-markup.mjs
cp docs/hosting/spfx/src/webparts/div296Calculator/markup.ts "$SPFX/src/webparts/div296Calculator/markup.ts"
```

**4. Build + verify locally:**

```bash
cd "$SPFX"
npm install
gulp build            # confirm a clean compile
gulp serve            # opens the local SharePoint Workbench to test the web part
```

**5. Package for production:**

```bash
gulp bundle --ship
gulp package-solution --ship
# → sharepoint/solution/div296-calculator.sppkg
```

**6. Hand off to IT.** Give the admin `div296-calculator.sppkg` plus
[`../IT-request.md`](../IT-request.md). They upload it to the **App Catalog**
(`/sites/appcatalog` → *Apps for SharePoint*) and trust it.

**7. Use it.** Once approved:
- **SharePoint page:** Edit → *+* → find **Division 296 Reset Calculator** (Advanced group) → add.
- **Teams tab:** in a channel, *+* → add the app / a Website tab pointing at the page hosting the web part.

## Keeping it in sync with the website

The web part reuses the site's logic. After any change under `web/`:
1. `node web/tests/parity.test.js` (must stay 23/23),
2. update `calcs.ts` if `web/calcs.js` changed, and `ui.ts` if `web/app.js` changed,
3. `node docs/hosting/spfx/build-spfx-markup.mjs` to refresh `markup.ts`,
4. rebuild + repackage (steps 4–5) and give IT the new `.sppkg`.
